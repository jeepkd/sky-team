1. Project Overview & Scope

What We're Building

  • A real-time browser-based implementation of Sky Team (base game)
  • Two players join via a room code — no accounts required
  • Either player can be replaced by a Claude AI agent (core feature)
  • Server-side dice rolling for fairness
  • Full hidden-information enforcement (die values hidden from opponent until reveal)

Tech Stack

  • Frontend: React 18 + TypeScript + Vite
  • Styling: Tailwind CSS + Framer Motion
  • Backend: Supabase (Postgres + Realtime + Edge Functions)
  • Auth: Supabase Anonymous Auth (no signup required)
  • AI Agent: Anthropic Claude API (claude-sonnet-4-6)
  • Hosting: Vercel (frontend) + Supabase (backend)

Core Constraints

  • Die values must be hidden from opponent during placement
  • Communication rules enforced: players can announce placement location, not value
  • All game logic validated server-side (edge functions)
  • AI agent must respect the same hidden-information constraints as humans
  • Graceful reconnection if a player drops

2. Sky Team Rules Reference (Dev Summary)

Key mechanics the codebase must encode. Refer to the official rulebook for edge cases.

Flow of a Round

flowchart LR A[Roll Phase\nServer rolls 4 dice\nper player — hidden] --> B[Planning Phase\nFree communication\nAllowed] B --> C[Placement Phase\nAlternate placing 1 die\nLocation OK, value FORBIDDEN] C --> D[Revelation\nAll dice visible\nsimultaneously] D --> E[Resolution\nApply module effects\nCheck for failure] E --> F{More rounds?} F -- Yes --> A F -- No --> G[Landing Phase\nBrakes / runway\nFinal checks]

Cockpit Modules (Base Game)

ModuleWho PlacesValidation RuleEffect
Axis (Joystick)Both (1 each)|Pilot - CoPilot| ≤ current turn limitMoves plane on approach track by result
Engine (L/R)Pilot=Left, CoPilot=RightBoth slots must be filled; sum within rangeAdjusts altitude / speed marker
FlapsEither (1 total)Value must be > last flap placed (ascending 1→4)Unlocks landing gear slot in later rounds
Landing GearEither (1 total)Value ≥ 3; only available after flaps roundGear deployed — required for landing
Radio (ATC)Either (optional)Value matches current traffic tokenClears traffic, avoids collision penalty
ConcentrationEither (optional)Any valueSpend to reroll one of your own dice
BrakesBoth (1 each)Sum ≤ max brake force allowedSlows plane on runway (landing phase only)
Coffee ☕EitherOne per player per gameCancel a bad die result (reroll or remove)
Critical enforcement point: During the Placement Phase players MUST NOT reveal die values — only the target module slot. Your frontend must prevent accidental disclosure (e.g., no "hover to see opponent value" until Revelation phase). The server must validate placements without broadcasting values to the wrong client.

3. System Architecture

Both human clients and the AI agent receive game state through the same Supabase Realtime channel. The AI agent is triggered by an Edge Function that fires after each placement event; it reads game state, calls the Claude API, then writes an action back through the standard place-die Edge Function.

7. Supabase Edge Functions

create-room

  • Generates unique 6-char room code
  • Creates rooms + games records
  • Assigns caller as Pilot or CoPilot based on query param
  • Returns { roomCode, gameId, role }

join-room

  • Validates room code, checks game is in LOBBY state
  • Assigns second player to remaining role
  • If joining as AI: sets is_ai=true, skips ready check
  • When both players joined → emits game_ready event

roll-dice

  • Uses crypto.getRandomValues() — no client influence
  • Inserts into dice_rolls (private to that player via RLS)
  • Fires for each player separately; after both rolled → phase → PLANNING
  • AI player rolls are triggered immediately by the function itself

place-die

  • Validates: correct player's turn, valid module, die exists in remaining pool
  • Validates module-specific rules (flap ascending, landing gear ≥3, etc.)
  • Inserts placements with revealed=false
  • Updates dice_rolls.remaining
  • If all 8 dice placed → triggers reveal-round

resolve-round

  • Atomically sets placements.revealed=true
  • Evaluates all module effects in dependency order
  • Checks failure conditions; updates game_state
  • Advances current_round or sets final status
  • Emits round_resolved event with full diff

ai-agent-tick

  • Triggered by placement events when it's the AI's turn
  • Reads full game state visible to AI role (respects RLS)
  • Builds structured prompt: rules + state + history + available actions
  • Calls Claude API with tools for structured action output
  • Executes returned action via internal place-die call
  • On PLANNING phase: generates a chat message via streaming

8. LLM Agent Design

Agent Architecture

flowchart TD A[Game Event: AI's turn to place] --> B[Build Context] B --> C[My dice values + remaining] B --> D[Board state: all modules\nwhat's placed so far] B --> E[Opponent placement history\nlocation only, no values] B --> F[Available actions list] B --> G[Round history + comms] C & D & E & F & G --> H[Claude API Call\nwith tool_use schema] H --> I{Response type?} I -- place_die action --> J[Execute place-die edge fn] I -- send_message --> K[Insert into messages table] I -- use_concentration --> L[Reroll die, update state] J & K & L --> M[Emit game_event] M --> N[Both clients update via Realtime]

System Prompt Strategy

  • Role: "You are the {role} of a commercial aircraft. You and your partner must cooperate to land safely."
  • Rules injection: Full module rules as structured JSON, not prose — LLM parses JSON more reliably
  • Communication constraint: Explicitly forbid revealing die values in messages: "You may say WHICH slot you intend to use, but NEVER the value."
  • Reasoning display: The reasoning field from place_die is shown to the human player as "AI is thinking..." — gives transparency without breaking rules
  • Context window: Include last 3 rounds of history + current round state. Truncate older history to stay within limits.
  • Temperature: Use 0.3–0.5 for placement decisions (consistent), 0.7 for planning messages (natural conversation)
Risk: The LLM may hallucinate invalid actions (e.g., place on a module that's already filled, or claim a die value it doesn't have). The ai-agent-tick edge function must validate all AI actions with the same game logic as human moves, and retry up to 2 times with the error message added to context before falling back to a safe default action.

  • diceSlice — my dice, selected die, placements visible to me

Game Scenario Runner /run-scenario Plays through a full scripted game scenario (dice values + placements predefined) and asserts the final game state. Catches regressions in resolve-round logic. High

RLS Policy Audit /audit-rls Connects to local Supabase and runs a suite of SQL queries that simulate both player roles, verifying hidden info is never leaked across role boundaries. Medium
AI Game Replay /replay-game Given a game_id, fetches all game_events and placements and steps through the game state turn by turn in the terminal. Debugging tool for completed games. Low

  • Cockpit UI: all base game modules rendered, accept die drops
  • DiceHand component: shows my dice, selection UX
  • Realtime sync: placement events, phase transitions
  • Revelation animation: dice flip reveal on all slots simultaneously
  • Approach track SVG: animated plane position updates
  • Win/loss detection and end-game screen
  • Exit criteria: Two humans can play a complete game from start to finish
Phase 3 — AI Agent (Weeks 6–7)
  • Edge function: ai-agent-tick with Claude API integration
  • System prompt design and iteration (use /test-agent skill)
  • Planning phase messages: AI communicates strategy without revealing dice
  • Placement phase: AI selects dice and modules autonomously
  • AI "thinking" streaming display in ChatPanel
  • Error recovery: retry on invalid AI action (up to 2x)

  • Exit criteria: Human can play a full game vs the AI agent at reasonable quality

  • Reconnection handling: rejoin game mid-round, restore state

  • Error boundary UX: toast notifications for edge function errors
  • Performance: memoize cockpit module renders, limit realtime payload size

13. Missing Components & Risks You Should Know About

These are significant pieces not yet addressed in the plan — each one is a real implementation decision.

Reconnection & Session Recovery

If a player's tab closes mid-game, how do they rejoin? Supabase anonymous sessions are browser-local. You need a strategy: either store session tokens in localStorage + validate on rejoin, or generate a one-time "rejoin code" shown to the player at game start.

Communication Rule Enforcement

The physical game relies on the honor system for the "don't say your die value" rule. In your digital version during PLACING phase, you could: (a) disable chat entirely, (b) allow free text but warn, or (c) run messages through a quick LLM filter that rejects value reveals. Option (c) adds latency. You need a policy decision here.

AI Agent Cost & Rate Limiting

Each AI turn makes a Claude API call. A full game is ~6 rounds × ~4 AI turns/round = ~24 API calls. At scale this adds up. Add per-room rate limiting in ai-agent-tick and expose the API key only server-side (Edge Function env var — never in frontend).

Cockpit Visual Design

The cockpit layout is a complex 2D board. You need a visual design decision: pixel-art style (Canvas/WebGL), SVG-based instrument panels, or a modern card-based layout. This will significantly affect build time. Recommend starting with cards, then upgrading the visuals once gameplay is solid.

Server-Side Timeout / Stuck Games

If one player goes offline during PLACING phase, the game hangs. Needs a server-side mechanism: either Supabase pg_cron polling for stale games, or a Deno cron within an Edge Function that force-resolves timed-out turns.

Dice Fairness Audit Log

Players may suspect dice are rigged. Consider storing the RNG seed (or at least a hash) with each dice_roll row so a player can independently verify the roll after the game. This is a trust feature, not just a debugging feature.

Approach Track Configuration

Different Sky Team airports have different approach track configs, traffic tokens, and special rules. Even if you're doing base game only, abstract the airport config into a JSON schema from day one. Hardcoding will hurt later.

AI Agent's "Persona"

The AI co-pilot/pilot needs a personality that makes it feel like a real crewmate — not just a move optimizer. Consider naming the AI, giving it a communication style (terse military, friendly civilian), and having it express uncertainty ("I'm not sure about the axis this round..."). This is a significant prompt engineering task that affects player experience more than any technical feature.
